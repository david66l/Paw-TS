## 二、Claude Code — 六层递进架构

### 2.1 架构概览

Claude Code 采用 **最复杂的分层策略**，从轻量级清理到全对话总结，共 6 层：

```
Layer 1: Microcompact      ← 清除旧工具结果（API cache_edits / 时间触发）
Layer 2: Session Memory    ← 后台持续记录关键信息到 markdown
Layer 3: Auto-Compact      ← 自动全对话总结（默认触发层）
Layer 4: Partial Compact   ← 手动定向总结（用户选择范围）
Layer 5: API-Native        ← Anthropic API 原生 context 编辑
Layer 6: Context Collapse  ← 实验性精细分段总结
```

### 2.2 各层详解

#### Layer 1: Microcompact（微压缩）

**目的**：不清除对话历史，仅删除旧工具结果的详细内容。

- **Cached microcompact**：利用 Anthropic API 的 `cache_edits` 服务端删除工具结果，不破坏 prompt cache
- **Time-based microcompact**：空闲一段时间后，只保留最近 N 个可压缩工具结果，其余替换为占位符
- 在每轮查询前运行，是**成本最低**的清理手段

#### Layer 2: Session Memory（会话记忆）

**目的**：后台持续提取关键信息到持久化 markdown 文件（`~/.claude/session-memory/`）。

- 每轮 API 调用后触发（post-sampling hook）
- 阈值：上下文增长 >= 5K tokens **且** >= 3 次工具调用
- 使用 forked subagent 读取当前笔记并增量更新
- 跟踪 `lastSummarizedMessageId` 避免重复处理
- 模板包含：Session Title, Current State, Task Specification, Files/Functions, Workflow, Errors, Learnings, Key Results, Worklog

#### Layer 3: Auto-Compact（自动压缩）

**目的**：当上下文接近上限时，对整个对话进行总结。

**触发条件**：
```
token_count >= effective_context_window - AUTOCOMPACT_BUFFER_TOKENS(13,000)
```

**流程**：
1. 先尝试 Session Memory Compaction（用 session memory 替代总结）
2. 若不足，fork 压缩 agent，传入完整对话 + summary prompt
3. Agent 输出结构化总结（见 2.3 格式）
4. 剥离 `<analysis>` 草稿，格式化 `<summary>` 块
5. 用 boundary marker + summary message 替换边界前的所有消息
6. 重新注入 deferred tools、agent listings、MCP instructions
7. **Prompt cache 共享**：压缩 fork 复用主对话的 prompt cache

#### Layer 4: Partial Compact（部分压缩）

用户手动选择一条消息，压缩 "from"（保留之前，总结之后）或 "up_to"（总结之前，保留之后）。

#### Layer 5: API-Native Context Management

直接使用 Anthropic API 原生功能：
- `clear_tool_uses_20250919` — 清除工具使用记录
- `clear_thinking_20251015` — 清除 thinking 块

默认：input token > 180K 时触发，保留最近 40K。

#### Layer 6: Context Collapse（实验性）

- 背景 ctx-agent 持续 staging 和 committing collapse
- 90% 阈值时非阻塞触发，95% 时阻塞触发
- 启用后抑制 auto-compact 避免竞态

### 2.3 总结格式（Auto-Compact）

```markdown
<primary_request>
<key_concepts>
<files_and_code>
<errors_and_fixes>
<problem_solving>
<user_messages>
<pending_tasks>
<current_work>
<next_steps>
```

### 2.4 关键阈值

| 阈值 | 值 | 说明 |
|------|-----|------|
| `AUTOCOMPACT_BUFFER_TOKENS` | 13,000 | 触发自动压缩的缓冲 |
| `MAX_OUTPUT_TOKENS_FOR_SUMMARY` | 20,000 | 总结输出 token 上限 |
| `POST_COMPACT_TOKEN_BUDGET` | 50,000 | 压缩后重新注入附件的预算 |
| `POST_COMPACT_MAX_FILES_TO_RESTORE` | 5 | 压缩后恢复的最大文件数 |
| `minimumMessageTokensToInit` | 10,000 | Session Memory 首次提取最小 token |
| `minimumTokensBetweenUpdate` | 5,000 | Session Memory 两次更新最小间隔 |
| `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES` | 3 | 连续失败熔断 |

## 三、OpenCode — 两层精简架构

### 3.1 架构概览

OpenCode 采用**最精简的策略**，只有两层：

```
Tier 1: Prune  ← 清除旧工具输出（轻量级）
Tier 2: Compaction  ← 全对话总结（重量级）
```

### 3.2 Tier 1: Prune（工具输出修剪）

在全对话压缩之前，先尝试轻量级优化：

- 从对话尾部向前遍历
- 保护最近 ~40K tokens 的工具调用
- 将更早的已完成工具输出标记为 "compacted"，替换为 `[Old tool result content cleared]`
- **PRUNE_MINIMUM = 20_000**：至少释放 20K tokens 才值得修剪
- **PRUNE_PROTECT = 40_000**：保护最近 40K tokens 的工具调用
- **PRUNE_PROTECTED_TOOLS = ["skill"]**：skill 工具永不修剪

### 3.3 Tier 2: Compaction（对话压缩）

**Overflow 检测**（`overflow.ts`）：
```typescript
const COMPACTION_BUFFER = 20_000;
usable = context_limit - max_output_tokens - COMPACTION_BUFFER;
// 或若 limit.input 存在: limit.input - reserved
```

当 `total_tokens >= usable` 时触发压缩。

**流程**：
1. **Tail 保护**（`select`）：
   - `tail_turns = 2`（默认）：保留最近 2 轮 user-assistant 对话
   - `preserve_recent_tokens = min(8000, max(2000, usable * 0.25))`：tail token 上限
2. **Summary 生成**：将 "head"（较老消息）发送给专用 compaction agent
3. **Anchored summaries**：保留之前的总结，将新历史合并进去（**不重新从头总结**）
4. **History 过滤**（`filterCompacted`）：用总结消息替换被压缩部分，保留 tail
5. **Auto-Continue**：压缩后注入合成用户消息 `"Continue if you have next steps..."`

### 3.4 总结格式

```markdown
## Goal
## Constraints & Preferences
## Progress
  - Done
  - In Progress
  - Blocked
## Key Decisions
## Next Steps
## Critical Context
## Relevant Files
```

### 3.5 关键阈值

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `compaction.auto` | `true` | 自动压缩开关 |
| `compaction.prune` | `true` | 工具输出修剪开关 |
| `compaction.tail_turns` | `2` | 保留最近轮数 |
| `compaction.preserve_recent_tokens` | `min(8K, max(2K, usable*0.25))` | Tail token 上限 |
| `compaction.reserved` | `min(20K, maxOutput)` | 保留缓冲 |
| `tool_output.max_lines` | `2000` | 工具输出最大行数 |
| `tool_output.max_bytes` | `50*1024` | 工具输出最大字节 |

### 3.6 设计亮点

- **Anchored summaries**：增量合并而非重新生成，节省 LLM 调用成本
- **Media stripping**：压缩时将图片附件替换为文本占位符 `[Attached image/png: filename]`
- **Overflow mode**：若压缩后仍溢出，重放上一轮用户输入（不带附件）

## 四、Hermes — 可插拔四阶段架构

### 4.1 架构概览

Hermes 采用**可插拔的 ContextEngine 架构**，默认实现为 `ContextCompressor`，分 4 阶段：

```
Phase 1: Prune old tool results      ← 单行摘要 + 去重（无 LLM 调用）
Phase 2: Determine boundaries        ← 保护 head + tail
Phase 3: Generate structured summary ← LLM 生成（辅助模型优先）
Phase 4: Assemble compressed list    ← 组装消息列表
```

### 4.2 Phase 1: Prune Old Tool Results

- 将旧工具输出替换为信息性单行摘要（如 `[terminal] ran 'npm test' -> exit 0, 47 lines output`）
- 去重相同工具结果（只保留最新完整副本）
- 从旧多模态内容中剥离图片
- 截断 assistant 消息中受保护 tail 之外的大型 tool_call 参数

### 4.3 Phase 2: Determine Boundaries

- **Protect head**：前 N 条消息（默认 `protect_first_n = 3`，即系统提示 + 首次交换）
- **Protect tail by token budget**：从尾部向前累加 token，直到达到 `tail_token_budget`（默认 ~20% threshold）
- **永不切割** tool_call/result 组
- **确保**最新的 user message 始终在 tail 中

### 4.4 Phase 3: Generate Structured Summary

- 使用**辅助模型**（auxiliary，更便宜更快），失败时回退到主模型
- **Iterative updates**：重新压缩时更新之前的总结，而非从头总结
- **结构化模板**（见 4.6）
- 序列化前进行 **redaction**，防止 secrets 泄漏到总结中
- 支持 **guided compression**（`/compress <focus_topic>`）：优先保留与主题相关的信息

### 4.5 Phase 4: Assemble Compressed Message List

- 在保护的 head 和 tail 之间插入总结消息
- 小心处理 role 交替，避免连续同 role 消息
- 清理孤立的 tool_call / tool_result 对
- 添加 `SUMMARY_PREFIX` 提示模型将总结视为背景参考而非活跃指令

### 4.6 总结模板

```markdown
## Active Task
## Goal
## Constraints & Preferences
## Completed Actions
## Active State
## In Progress
## Blocked
## Key Decisions
## Resolved Questions
## Pending User Asks
## Relevant Files
## Remaining Work
## Critical Context
```

### 4.7 关键阈值

| 阈值 | 默认值 | 说明 |
|------|--------|------|
| `compression.threshold` | `0.50` | 上下文窗口的 50% 触发压缩 |
| `compression.target_ratio` | `0.20` | tail 保护的阈值比例 |
| `compression.protect_last_n` | `20` | 最少保护 tail 消息数 |
| `_MIN_SUMMARY_TOKENS` | `2,000` | 总结输出最小 token |
| `_SUMMARY_RATIO` | `0.20` | 压缩内容中分配给总结的比例 |
| `_SUMMARY_TOKENS_CEILING` | `12,000` | 总结 token 绝对上限 |
| `_IMAGE_TOKEN_ESTIMATE` | `1,600` | 每张图片固定 token 成本 |
| `_CHARS_PER_TOKEN` | `4` | 粗略字符/token 比 |
| `_SUMMARY_FAILURE_COOLDOWN_SECONDS` | `600` | 总结失败冷却时间 |

### 4.8 Anti-Thrashing & 安全机制

1. **Anti-thrashing**：连续两次压缩各节省 <10%，则跳过压缩
2. **Summary failure cooldown**：30-60s 瞬态错误，600s "no provider configured"
3. **Aux model fallback**：辅助模型失败（404/503/超时/JSON 解析错误）立即回退主模型
4. **Static fallback**：LLM 总结完全失败时插入 fallback marker
5. **Context probing**：上下文长度错误时逐级降级 `[256K, 128K, 64K, 32K, 16K, 8K]`
6. **Max compression attempts**：错误恢复路径最多 3 次尝试

### 4.9 Memory 集成

- `MemoryManager.on_pre_compress(messages)` 在压缩丢弃上下文前调用
- Memory provider 可实现在压缩前提取关键信息持久化

## 五、设计模式对比

### 5.1 触发策略

| 项目 | Claude Code | OpenCode | Hermes |
|------|-------------|----------|--------|
| **触发时机** | 每轮前检查 | 每轮前 overflow 检查 | Preflight + Post-response |
| **触发阈值** | `context - 13K` | `context - output - 20K` | `context * 50%` |
| **保护 Head** | 前 N 消息 | 无 | 前 3 消息 |
| **保护 Tail** | Token budget | 2 轮 + token cap | Token budget (~20%) |
| **轻量级清理** | Microcompact (2 种) | Prune (工具输出) | Phase 1 Prune |

**分析**：
- Claude Code 的触发最保守（13K buffer），OpenCode 最激进（20K + output），Hermes 居中（50% 窗口）
- Hermes 的 Head 保护（系统提示 + 首轮）对保持任务上下文最有帮助
- OpenCode 的 `tail_turns=2` 更直观但不如 token-based 灵活

### 5.2 总结生成

| 项目 | Claude Code | OpenCode | Hermes |
|------|-------------|----------|--------|
| **总结模型** | Fork subagent | Compaction agent | Auxiliary model |
| **增量更新** | Session Memory | Anchored summaries | Iterative updates |
| **失败回退** | Circuit breaker (3次) | Overflow mode | Static fallback + cooldown |
| **Cache 复用** | Prompt cache 共享 | 无 | 无 |

**分析**：
- Claude Code 的 prompt cache 共享是显著优化，压缩时无需重新构建整个 prompt
- OpenCode 的 anchored summary 设计优雅，适合频繁压缩场景
- Hermes 的 auxiliary model 降级策略最完善

### 5.3 总结格式丰富度

| 维度 | Claude Code | OpenCode | Hermes |
|------|-------------|----------|--------|
| **任务/目标** | ✅ Primary Request | ✅ Goal | ✅ Active Task + Goal |
| **进度追踪** | ✅ Current Work / Pending | ✅ Progress (Done/In Progress/Blocked) | ✅ Completed / In Progress / Blocked |
| **决策记录** | ✅ Key Decisions | ✅ Key Decisions | ✅ Key Decisions |
| **文件引用** | ✅ Files and Code | ✅ Relevant Files | ✅ Relevant Files |
| **错误记录** | ✅ Errors & Fixes | ❌ | ❌ |
| **学习/经验** | ✅ Learnings | ❌ | ❌ |
| **用户消息** | ✅ User Messages | ❌ | ❌ |
| **下一步** | ✅ Next Steps | ✅ Next Steps | ❌ |
| **约束/偏好** | ❌ | ✅ Constraints & Preferences | ✅ Constraints & Preferences |
| **待处理问题** | ✅ Pending Tasks | ❌ | ✅ Pending User Asks |

**分析**：
- Claude Code 的总结格式最全面（10+ 个维度），适合长对话和复杂任务
- Hermes 的格式次之，但包含 "Pending User Asks" 等实用维度
- OpenCode 最精简，但 "Progress" 的三态分类（Done/In Progress/Blocked）非常实用

### 5.4 Token 管理

| 项目 | Claude Code | OpenCode | Hermes |
|------|-------------|----------|--------|
| **估算方式** | API 精确 + 启发式 | `length / 4` | `length / 4` + 图片固定值 |
| **上下文探测** | 1M / 200K 切换 | 无 | `[256K..8K]` 逐级降级 |
| **用户预算** | `+500k` 语法 | 无 | 无 |
| **工具 schema 计入** | ✅ | ❌ | ✅ |

## 六、对 paw-ts 的借鉴建议

### 6.1 最小可行方案（MVP）

参考 **OpenCode** 的两层设计，paw-ts 可以先实现：

1. **Prune 层**：清除旧工具输出（保留最近 N 轮），替换为单行摘要
2. **Compact 层**：当对话超过阈值时，fork 一个 summarization agent 生成总结

### 6.2 推荐阈值

```typescript
const COMPACTION_THRESHOLD = 0.75;        // 75% 上下文窗口触发
const COMPACTION_BUFFER = 8_000;          // 8K token 缓冲
const TAIL_TURNS = 2;                     // 保留最近 2 轮
const PRUNE_PROTECT_TOKENS = 20_000;      // 保护最近 20K tokens 的工具结果
```

### 6.3 推荐总结格式

结合三者的优点，采用以下结构：

```markdown
## Goal
## Progress
  - ✅ Done
  - 🔄 In Progress
  - ⛔ Blocked
## Key Decisions
## Relevant Files
## Errors & Fixes
## Next Steps
## Pending Questions
```

### 6.4 关键设计决策

| 决策 | 建议 | 理由 |
|------|------|------|
| **增量 vs 全量总结** | 增量（Anchored） | 节省 LLM 调用成本，参考 OpenCode |
| **Head 保护** | 保护系统提示 + 首轮用户消息 | 保持任务上下文，参考 Hermes |
| **Tail 保护** | Token-based + 轮数兜底 | 比纯轮数更灵活，参考 Claude Code |
| **失败处理** | 3 次熔断 + 静态 fallback | 避免无限重试，参考 Claude Code + Hermes |
| **总结模型** | 主模型（无 auxiliary） | paw-ts 暂无需多模型支持 |
| **Anti-thrashing** | 连续节省 <15% 则跳过 | 避免无效压缩，参考 Hermes |

### 6.5 实现优先级

1. **P0**：Prune 层（工具输出清理）— 最简单，收益最大
2. **P0**：Overflow 检测 + 基础 Compaction — 防止上下文溢出
3. **P1**：增量总结（Anchored）— 提升长对话体验
4. **P1**：Head/Tail 保护策略 — 保持关键上下文
5. **P2**：结构化总结模板 — 提升总结质量
6. **P2**：Anti-thrashing + 熔断机制 — 稳定性
