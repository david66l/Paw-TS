## 架构层面差距

| 维度 | Paw-TS | Claude Code |
|------|--------|-------------|
| **包组织** | Monorepo (packages + apps) | 单仓库扁平结构 |
| **依赖注入** | 构造函数传参 | 更复杂的 context/hook 系统 |

| **错误处理** | 基础 PawError | 多层 error + recovery |
| **配置系统** | 基础 settings schema | 多层 config + migration |

## 建议的追赶路径

### Phase 1 — 核心补齐（2-3周）

1. **Context 管理** — 添加 session history + message 类型丰富化
2. **工具扩展** — 添加 GlobTool, GrepTool, FileEditTool（非全量覆盖）
3. **Streaming 完善** — tool use 从 stream 中实时提取

### Phase 2 — 生态建设（2-4周）

1. **MCP 支持** — 至少 client 端，连接外部 tools
2. **Skills 框架** — 基础 skill 加载 + 执行
3. **WebFetch/WebSearch** — 网络能力

### Phase 3 — 体验提升（1-2周）

1. **TUI 丰富** — 输入历史、快捷键、更好的状态显示
2. **命令系统** — 添加 commit、config 等实用命令
3. **Cost tracking** — token 使用量追踪

### Phase 4 — 基础设施（可选）

1. Feature flags
2. OpenTelemetry
3. Plugin marketplace

## 关键洞察

**好消息：** Paw-TS 的 monorepo 架构比 Claude Code 的扁平结构更适合长期维护。packages 的边界（core, harness, models, agent, store, workspace, settings）是合理的。

**核心问题：** Paw-TS 目前是一个**骨架** — orchestrator + 5 tools + 基础 TUI。Claude Code 是一个**完整产品** — 包含产品化的所有细节（auth、telemetry、cost tracking、团队协作、MCP 生态）。

**务实的目标：** 不需要达到 50 万行代码。重点是补齐 **P0-P2** 的核心能力（context、tools、MCP），让 Paw-TS 从"能跑"变成"好用"。
