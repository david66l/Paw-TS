## 二、面试中指出的未修复项

### P0 — 安全边界仍需加固

#### 1. Shell guard：黑名单 → AST 解析（未做）

**面试原话**：
> "shell guard 只能降低风险，不能证明安全。如果只是词法黑名单，很容易漏。大厂面试官会看你有没有意识到这一点。"

> "企业级我会用 shell AST 解析器，比如 tree-sitter-bash，把命令解析成结构化节点，递归检查管道、重定向、命令替换、子 shell、脚本参数。"

**当前状态**：`shell-guard.ts` 仍是正则 + 关键词匹配。补了 `find -delete`、`python -c` 中的 `shutil.rmtree`、`| curl` 管道外传，本质没有变。

**升级方案**：
- 引入 tree-sitter-bash，解析 AST
- 递归检查：管道每段、重定向目标、命令替换 `${}`、子 shell `$(...)`
- 按节点类型分级：纯读 / 敏感读 / 写 / 网络外传 / 破坏性
- shell guard 定位从"拦截"改为"风险识别 + 触发审批"

---

#### 2. 面试中描述的分层安全模型未实现

**面试原话**（追问"怎么升级成可靠执行安全模型"的回答）：
> "第一层权限模型（safe read / sensitive read / write / network / destructive）。第二层命令解析（tree-sitter-bash AST）。第三层文件系统策略（shell 默认只读 workspace，写入走 staging 目录，受控 apply 阶段展示 diff 用户确认）。第四层沙箱默认开启。第五层网络控制（白名单域名，严格模式 --network none）。第六层审批和审计（风险打分，授权范围限定）。"

**当前状态**：这些全部在面试口述中存在，但**代码中均未实现**。如果面试官说"给我看代码"，会露馅。

**升级方案**：至少实现三层：
1. **命令分级**：在 `classifyShellCommand` 基础上扩展为五级（safe_read / sensitive_read / write / network / destructive）
2. **staging 写入**：shell 不直接写 workspace，输出到 `staging/` 目录，用户确认后 `apply_patch`
3. **审批风险打分**：不是简单的布尔 yes/no，结合命令类型、目标路径、网络目标、参数模式算分

---

#### 3. Docker 沙箱仍是可选，不是默认

**面试原话**：
> "个人项目里 Docker 可以是可选兜底，企业级必须默认开启。"

**当前状态**：`resolve-shell-sandbox.ts` 支持 Docker 沙箱配置，但需要用户显式传参开启。默认不开启。

**升级方案**：默认开启 Docker 沙箱模式，至少对 `run_shell` 强制执行。提供 `--no-sandbox` 显式关闭选项。

---

### P1 — Agent Loop 可靠性

#### 4. auto_nudge 重试无指数退避

**面试原话**（追问"工具执行失败后 Agent loop 怎么恢复"）：
> "E_RETRY：网络抖动、429、临时文件锁。执行器内部自动重试，带指数退避，不一定暴露给模型。"

**当前状态**：`orchestrator.ts` 中模型调用有 `callModelWithRetry`（指数退避 + jitter），但工具执行层没有重试逻辑。工具失败直接写回上下文让模型自己修。

**升级方案**：在 `executeTool` 返回 `E_RETRY` 错误码时，执行器内部自动重试（指数退避，最多 3 次），不暴露给模型。只在重试全部失败后才写回上下文。

---

#### 5. 工具调用协议未升级为原生 function calling 优先

**面试原话**（追问"企业级工具调用协议"）：
> "企业级默认应该走模型厂商的原生结构化工具调用。Orchestrator 不需要从自然语言里猜模型意图。"

**当前状态**：`parse-agent-action.ts` 仍从文本里扫 JSON。虽然加了 `knownTools` 白名单，但本质没变——没有优先使用 OpenAI tool_calls / Anthropic tool_use block。

**升级方案**：
- `invokeModel` 时传入 `tools` 参数（registry.ts 已有 `toolDefinitions()`）
- 响应解析优先读取 API 返回的 `tool_calls` 结构化字段
- 文本扫描仅作为不支持原生 tool calling 的 provider 的 fallback

---

### P2 — 微调/模型能力

#### 6. 目前使用通用模型，对 Paw Code 工具格式无专门优化

**面试中隐含的期望**：一个合格的 Coding Agent 项目应该考虑过"如何让模型更擅长输出你的工具格式"。当前全靠 system prompt 约束 + auto_nudge 兜底。

**升级方案**：这就是微调计划的目标——QLoRA Qwen3-8B，提升 tool_call 格式遵循率。

## 三、面试回答层面的改进建议

### 口径问题

Codex 一面总结明确指出三个扣分点之一：
> "口径不稳定。Paw Code 里哪些写完了、哪些是设计稿，必须统一话术。"

**建议**：准备一张"能力矩阵"表，面试时直接说清楚：

| 能力 | 状态 | 代码位置 |
|------|------|---------|
| ReAct+Plan 混合 Loop | ✅ 完整实现 | `orchestrator.ts` |
| 三层上下文治理 | ✅ 完整实现 | `context-pruner.ts` + `compression-agent.ts` + `context-manager.ts` |
| 跨会话记忆 | ✅ 完整实现 | `memory-retriever.ts` + `memory-extraction-agent.ts` |
| Path guard + realpath | ✅ 已修复 | `path-guard.ts` |
| Tool call schema 校验 | ⚠️ 基础版 | `registry.ts:validateArgs()` |
| Shell guard AST 解析 | ❌ 设计稿 | 未实现 |
| 分层安全模型 | ❌ 设计稿 | 未实现 |
| 多 provider adapter | ⚠️ 有 OpenAI 兼容，不完整 | `models/` |
| 微调模型 | 📋 计划中 | `微调与推理优化计划.md` |

### 项目定位话术

Codex 评价中最强的一段（建议保留）：
> "Paw Code 不是线上产品，它证明的是我对 Agent 底层执行链路有系统性理解。和只做过标准 RAG API 集成的候选人相比，我的优势是能深入到上下文治理、工具协议、安全执行、错误恢复这些 Agent 落地难点。"
