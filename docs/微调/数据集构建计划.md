# 微调数据集构建计划

> 基于 Paw Code `.paw/sessions/` 中的 ~90 个 session、~75K 行原始日志

---

## 0. 现状摸底

```
原始数据：90 个 session，75,467 行 JSONL
可提取字段：
  - agent.action → { type: "tool_call", tool, args }    ← 训练标签
  - tool.result  → { tool, ok, summary }                 ← 上下文
  - run.started  → { goal }                             ← 用户意图
  - run.completed → { status, message }                 ← 任务结果
```

**实际可用的量**：大部分 session 是调试 session（打招呼、无工具调用），需要先过滤。预估 30-40% 包含有效 tool_call，即 ~30-35 个 session 可用。

---

## 1. 数据集 A：Tool Calling 单步格式（目标 ≥1000 条）

### 1.1 从日志提取（预估 500-700 条）

```bash
bun run scripts/extract-training-data.ts --stage tool_call
```

提取逻辑：

```
遍历每个 session：
  1. 读 run.started → goal 字段作为 user instruction
  2. 读所有 agent.action 事件，取 tool + args
  3. 每对 (goal, tool_call) 生成一条训练样本
  4. 跳过 goal 无意义或 tool 调用失败的样本
```

输出格式（JSONL）：

```json
{"messages": [
  {"role": "system", "content": "You are Paw Code, a coding agent. Reply with a single JSON object containing 'tool' (one of the registered tool names) and 'args' (arguments object)."},
  {"role": "user", "content": "查看当前项目结构"},
  {"role": "assistant", "content": "{\"tool\":\"workspace.brief\",\"args\":{\"path\":\".\",\"max_files\":80}}"}
]}
{"messages": [
  {"role": "system", "content": "..."},
  {"role": "user", "content": "读取 package.json 文件"},
  {"role": "assistant", "content": "{\"tool\":\"workspace.read_file\",\"args\":{\"path\":\"package.json\"}}"}
]}
```

### 1.2 模板合成补充（预估 500+ 条）

日志覆盖不了所有工具的所有参数组合。需要模板生成补全：

```python
# generate_tool_call_templates.py

TEMPLATES = {
    "workspace.read_file": [
        {"user": "读取 src/main.ts", "args": {"path": "src/main.ts"}},
        {"user": "读取第 50 行开始的 config.py", "args": {"path": "config.py", "offset": 50}},
        {"user": "看下 README.md 的前 100 行", "args": {"path": "README.md", "limit": 100}},
        # ... 20 种变体
    ],
    "workspace.list_dir": [
        {"user": "列出当前目录的文件", "args": {"path": "."}},
        {"user": "递归查看 src 目录下有什么", "args": {"path": "src", "recursive": True}},
        # ... 10 种变体
    ],
    # ... 每个工具 10-20 种变体
}
```

按计划中覆盖要求的表格生成，确保每个工具类别的样本量达标。

### 1.3 格式修正样本（30 条）

```
用户消息里包含错误 JSON → assistant 输出修正后的正确格式

示例：
  user: "我之前写的 {\"tool\":\"read_file\",\"args\":{\"path\":\"x\"} 对吗？你帮我改成正确的"
  assistant: "{\"tool\":\"workspace.read_file\",\"args\":{\"path\":\"x\"}}"
```

这类样本训练模型**识别错误并自动修正**，而不是学习输出错误格式。

## 2. 数据集 B：工程行为工作流（目标 50-100 条）

### 2.1 从日志筛选完整轨迹（预估 15-30 条）

筛选条件：

```
必须包含的 event 序列：
  ✅ 至少一次 list_dir / brief / glob（观察）
  ✅ 至少一次 read_file（阅读）
  ✅ 至少一次 write_file / edit_file / apply_patch（修改）
  ✅ 至少一次 run_shell（验证）
  ✅ 最终 final_answer（完成）

排除条件：
  ❌ 没有观察阶段的（直接上来就改）
  ❌ 没有验证阶段的（改完就说完成）
  ❌ 最终状态是 failed / error 的
```

格式化：将整个 session 的多轮对话转换为 ChatML 格式：

```json
{"messages": [
  {"role": "system", "content": "<Paw Code system prompt>"},
  {"role": "user", "content": "给 paw-ts 项目添加一个 health check 端点"},
  {"role": "assistant", "content": "{\"tool\":\"workspace.list_dir\",\"args\":{\"path\":\".\"}}"},
  {"role": "tool", "content": "[Tool workspace.list_dir completed]\npackages/ apps/ package.json ...", "name": "workspace.list_dir"},
  {"role": "assistant", "content": "{\"tool\":\"workspace.read_file\",\"args\":{\"path\":\"packages/core/src/index.ts\"}}"},
  {"role": "tool", "content": "[Tool workspace.read_file completed]\nexport * from './health' ...", "name": "workspace.read_file"},
  {"role": "assistant", "content": "{\"tool\":\"workspace.write_file\",\"args\":{\"path\":\"packages/core/src/health.ts\",\"content\":\"export function check() { return { ok: true }; }\"}}"},
  {"role": "tool", "content": "[Tool workspace.write_file completed]\npackages/core/src/health.ts (45 bytes)", "name": "workspace.write_file"},
  {"role": "assistant", "content": "{\"tool\":\"workspace.run_shell\",\"args\":{\"command\":\"bun test\"}}"},
  {"role": "tool", "content": "[Tool workspace.run_shell completed]\nexit 0\nAll tests passed.", "name": "workspace.run_shell"},
  {"role": "assistant", "content": "{\"action\":\"final_answer\",\"summary\":\"已添加 health check 端点，测试通过\"}"}
]}
```

### 2.2 高质量模型生成种子轨迹（预估 30-50 条）

日志里符合"完整观察→修改→测试→完成"的工作流不够。用 Claude Opus + Paw Code 自动生成：

```
对每个任务模板：
  1. 用 Claude Opus 作为 Paw Code 的模型
  2. 给它一个具体任务
  3. 收集完整的多轮轨迹
  4. 只保留"正确行为模式"的轨迹（观察→修改→测试→完成）
  5. 人工 review 确保每步合理
```

任务模板示例：

```
1. 在 paw-ts 中创建一个新的 core 工具函数
2. 给某个包添加单元测试
3. 修复一个类型错误
4. 重构一个函数并保证测试通过
5. 搜索并替换项目中的某个模式
6. 添加一个新的 CLI 命令
...
```

### 2.3 关键技术细节

**tool role 的处理**：
- 在 ChatML 训练格式中，需要追加 `tool` role 消息承载工具返回结果
- Qwen3-8B-Instruct 原生支持 `role: "tool"` + `name` 字段
- 训练时也按这个格式喂

**system prompt**：
- 阶段 A 用简化版 system prompt（只含工具列表和格式要求）
- 阶段 B 用完整版 system prompt（包含安全约束、行为规范）

**数据分割**：
```
阶段 A 训练集：1000 条（800 日志提取 + 200 模板合成）
阶段 A 验证集：200 条
阶段 B 训练集：80 条（含 Claude 生成）
阶段 B 验证集：20 条
```

## 3. 执行脚本

### 3.1 日志提取脚本

```bash
# paw-ts/scripts/extract-training-data.ts
# 输入：.paw/sessions/*.jsonl
# 输出：training_data/stage_a_raw.jsonl + training_data/stage_b_raw.jsonl
bun run scripts/extract-training-data.ts
```

逻辑：

```
1. 遍历 .paw/sessions/*.jsonl
2. 每个 session：
   a. 收集 run.started.goal
   b. 收集所有 agent.action（tool name + args）
   c. 判断是否是完整工作流（stage B 候选）
   d. 生成对应格式的训练样本
3. 输出两个文件
```

### 3.2 模板合成脚本

```bash
# 补充 stage A 数据到 1000+
python scripts/generate_tool_call_templates.py \
    --output training_data/stage_a_templates.jsonl \
    --count 500
```

### 3.3 质量检查

```bash
python scripts/validate_training_data.py \
    --stage-a training_data/stage_a_*.jsonl \
    --stage-b training_data/stage_b_*.jsonl \
    --check
```

检查项：
- tool name 是否都在注册表中
- JSON 是否可解析
- stage B 轨迹是否包含必需的阶段
- 是否有重复样本
- system prompt 是否一致
