## 2. 微调方案

### 2.1 训练环境

```bash
# 云 GPU 环境搭建
pip install torch transformers peft bitsandbytes accelerate datasets
pip install vllm
```

### 2.2 QLoRA 配置

```python
from transformers import BitsAndBytesConfig, AutoModelForCausalLM
from peft import LoraConfig, get_peft_model

# 4bit 量化
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,
)

model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen3-8B-Instruct",
    quantization_config=bnb_config,
    device_map="auto",
    trust_remote_code=True,
)

# LoRA 配置
lora_config = LoraConfig(
    r=16,                    # rank：16 是社区最佳实践
    lora_alpha=32,           # 缩放因子：2×r
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)

model = get_peft_model(model, lora_config)
# 可训练参数：约 0.5% of 7B ≈ 35M
```

### 2.3 训练参数

```python
from transformers import TrainingArguments

training_args = TrainingArguments(
    per_device_train_batch_size=4,       # 24GB 可以跑 4
    gradient_accumulation_steps=8,        # 有效 batch = 32
    num_train_epochs=3,
    learning_rate=2e-4,
    warmup_ratio=0.1,
    lr_scheduler_type="cosine",
    logging_steps=10,
    save_steps=100,
    eval_steps=100,
    bf16=True,
    gradient_checkpointing=True,
    optim="adamw_8bit",
    output_dir="./paw-qwen3-lora",

```

### 2.4 显存预算（加 MLP 层后）


## 3. 训练数据构造

### 3.1 方向 A：Tool Calling 格式微调（≥1000 条）

**数据来源**：Paw Code 运行日志中 `parseAgentActionsFromModelText` 成功解析的 tool_call。日志不够时先用模板合成一批，再混真实日志。

**数据格式**：

```json
{
  "messages": [
    {"role": "system", "content": "<Paw Code system prompt>"},
    {"role": "user", "content": "读取 src/main.ts 文件"},
    {"role": "assistant", "content": "{\"tool\":\"workspace.read_file\",\"args\":{\"path\":\"src/main.ts\"}}"}
  ]
}
```

**覆盖要求**：

| 工具类别 | 样本数 | 说明 |
|---------|--------|------|
| read_file | 120 | 各种路径格式 |
| list_dir | 70 | 带/不带 recursive |
| grep/search | 100 | pattern + file_pattern |
| write_file/edit_file | 120 | 包含 content/patch |
| run_shell | 100 | 安全命令 + 分类 |
| run_agent | 50 | 子 Agent 调度 |
| todo_write | 30 | 任务管理 |
| final_answer | 80 | 任务完成 |
| plan_update | 30 | 计划变更 |
| ask_user | 20 | 向用户提问 |
| 格式修正样本 | 30 | 用户消息中出现错误 JSON，assistant 输出修正后的正确格式 |

**❌ 不要做的事**：

- 不要让 assistant 角色输出错误 JSON 作为训练样本——模型会学坏
- 正确的做法：用户/历史中出现错误输出 → assistant 输出修正后的正确格式

### 3.2 方向 B：工程行为偏好（50-100 条完整轨迹）

**数据来源**：Paw Code 成功完成的端到端 session 日志。不够时用 Paw Code + 高质量模型（Claude Opus）生成种子轨迹。

**每条轨迹必须包含**：

```
1. 用户提出任务
2. 模型先调用 list_dir/search 了解项目结构
3. 模型调用 read_file 读取相关文件
4. 模型调用 edit_file/write_file 进行修改
5. 模型调用 run_shell 运行测试
6. 如果测试失败 → 回到步骤 3（读报错→修复→重测）
7. 测试通过后调用 final_answer 总结
```

**强制要求**：
- 每条轨迹不能跳过步骤 2（先观察再动手）
- 每条轨迹必须有步骤 5（改完必须测试）
- 删除所有"没测试就说完成"的轨迹

**强制要求**：
- 每条轨迹不能跳过步骤 2（先观察再动手）
- 每条轨迹必须有步骤 5（改完必须测试）
- 删除所有"没测试就说完成"的轨迹

## 4. 训练流程

### 4.1 预处理

```bash
# 从 Paw Code 日志提取训练数据
cd paw-ts
bun run scripts/extract-training-data.ts
# 输出：training_data/tool_calls.jsonl + training_data/workflows.jsonl
```

### 4.2 Sequential SFT（不做 LoRA 合并）

**❌ 不要**：两个 LoRA adapter 分别训练再合并——不同 adapter 学到的东西可能冲突。

**✅ 正确**：阶段 A 先训，评估通过后在同一 adapter 上 continue SFT 阶段 B：

```bash
# 阶段 A：Tool Calling 格式 SFT
python train_tool_calling.py \
    --data training_data/tool_calls.jsonl \
    --model Qwen/Qwen3-8B-Instruct \
    --output ./paw-qwen3-lora-stageA

# 评估通过后 → 阶段 B：在同一个 LoRA 上 continue SFT
python train_workflow.py \
    --data training_data/workflows.jsonl \
    --model Qwen/Qwen3-8B-Instruct \
    --resume_from ./paw-qwen3-lora-stageA \
    --output ./paw-qwen3-lora-final
```

最终产物：**一个 LoRA adapter**，包含两个阶段的训练成果。

### 4.3 合并 LoRA 到基座模型（vLLM 部署前）

vLLM 加载 merged model 比挂 LoRA adapter 更简单：

```bash
# 将 LoRA 权重合并到基座模型
python merge_lora.py \
    --base Qwen/Qwen3-8B-Instruct \
    --lora ./paw-qwen3-lora-final \
    --output ./paw-qwen3-lora-final-merged

# 合并后的目录结构：
# ./paw-qwen3-lora-final-merged/
#   ├── config.json
#   ├── model-00001-of-00002.safetensors
#   ├── model-00002-of-00002.safetensors
#   └── tokenizer.json
```

合并后是标准的 HuggingFace 格式，vLLM 直接加载：`vllm serve ./paw-qwen3-lora-final-merged`。

## 6. vLLM 推理部署

### 6.1 启动配置

```bash
vllm serve ./paw-qwen3-lora-final-merged \
    --host 0.0.0.0 \
    --port 8000 \
    --max-model-len 8192 \
    --max-num-seqs 4 \
    --gpu-memory-utilization 0.90 \
    --enable-prefix-caching \
    --dtype bfloat16
```

### 6.2 参数解释

| 参数 | 值 | 原因 |
|------|-----|------|
| max-model-len | 8192 | system prompt + 工具列表 + 用户任务 + 观察结果 + 报错信息，2K 容易爆 |
| max-num-seqs | 4 | 单用户 Agent，4 并发够用 |
| gpu-memory-utilization | 0.90 | 留 10% buffer 防 OOM |
| enable-prefix-caching | ✅ | system prompt 每轮复用，prefill 省 30%+ |

### 6.3 结构化输出（Structured Outputs）

**注意版本兼容**：vLLM v0.12.0 起旧的 `guided_json` / `guided_decoding_backend outlines` 已移除，统一使用 `structured_outputs`。

```python
# 请求中通过 extra_body 传入 JSON Schema 约束
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-needed",
)

# Paw Code 工具调用约束
tool_call_schema = {
    "type": "json_schema",
    "json_schema": {
        "name": "paw_tool_call",
        "strict": True,
        "schema": {
            "type": "object",
            "properties": {
                "tool": {
                    "type": "string",
                    "enum": [
                        "workspace.read_file", "workspace.list_dir",
                        "workspace.search", "workspace.write_file",
                        "workspace.edit_file", "workspace.glob",
                        "workspace.grep", "workspace.run_shell",
                        "workspace.web_fetch", "workspace.web_search",
                        "workspace.todo_write", "workspace.notebook_edit",
                        "workspace.brief", "workspace.git_status",
                        "workspace.git_log", "workspace.git_diff",
                        "workspace.run_agent", "workspace.run_skill",
                        "workspace.lsp", "workspace.apply_patch",
                        "workspace.symbol_search",
                        "memory.list", "memory.read",
                    ]
                },
                "args": {"type": "object"}
            },
            "required": ["tool", "args"],
            "additionalProperties": False
        }
    }
}

response = client.chat.completions.create(
    model="paw-qwen3",
    messages=[
        {"role": "system", "content": "<Paw Code system prompt>"},
        {"role": "user", "content": "读取 src/main.ts"}
    ],
    extra_body={"structured_outputs": tool_call_schema}
)
```

**关键点**：
- `additionalProperties: False` — 不允许模型输出 schema 以外的字段，进一步降低格式错误
- `strict: True` — vLLM 会保证输出 100% 符合 schema
- 如果 vLLM 版本较老，暂时保留旧的 `guided_json` 作为 fallback

### 6.4 验证

```bash
# Python SDK（推荐）：用 extra_body 传入 structured_outputs
```

### 7.2 Tool Call 协议升级

vLLM 的 `structured_outputs` 保证输出是合法 JSON Schema。升级 parse-agent-action.ts：

```typescript
// 优先读取 API 响应的 tool_calls 结构化字段（原生 tool_use block）
if (response.choices[0].message.tool_calls) {
  return response.choices[0].message.tool_calls;
}
// vLLM structured_outputs 返回的 content 已是合法 JSON，不需要从文本里扫
const content = response.choices[0].message.content;
try {
  const parsed = JSON.parse(content);
  // structured_outputs 已保证 tool 在注册表里、required 字段齐全
  if (parsed.tool) {
    return [{ type: "tool_call", tool: parsed.tool, args: parsed.args }];
  }
} catch {
  // fallback：自由文本 JSON 扫描（兼容无 structured_outputs 的 provider）
  return parseAgentActionsFromModelText(content);
}
```

## 9. 面试话术

**微调部分**：

> "我用 QLoRA 微调了 Qwen3-8B-Instruct，分两个阶段做 Sequential SFT。阶段 A 训 Tool Calling 格式——1000+ 条工具调用样本，把 JSON 格式遵循率从 ~85% 提到 ≥95%，工具名正确率到 ≥98%。阶段 B 在同一个 LoRA 上 continue SFT 工程行为——50+ 条端到端轨迹，训练模型先观察再动手、改完跑测试、失败读报错。不做两个 LoRA 合并——那个会冲突。训练全程 RTX 4090D，QLoRA 4bit，可训练参数 ~35M，两阶段各跑几个小时。"

**推理优化部分**：

> "推理用 vLLM，三个关键配置。第一，Prefix Caching——system prompt 每轮复用 KV cache，prefill 省 40%。第二，Structured Outputs——用 JSON Schema + strict mode 约束输出，`additionalProperties: false` 保证模型只输出注册表里的工具名和合法参数格式。不是正则后处理——生成每个 token 时只允许 schema 范围内的候选，显著降低了非法 JSON 和工具名出错的概率。第三，PagedAttention——KV cache 显存利用率翻倍。三层叠加，单次 tool call 从 3 秒降到 1 秒左右。"

**数据驱动闭环**：

> "整个微调的评估指标不是 loss 曲线——是 Tool Calling 解析成功率、auto_nudge 触发次数、改完是否跑测试。指标直接和 Agent 的行为质量挂钩，不搞虚的。"
