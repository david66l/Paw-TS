# Claude Code / OpenCode / Hermes 记忆机制解析

> 分析日期：2026-05-12

---

## 一、总体对比

| 维度 | Claude Code | OpenCode | Hermes |
|------|-------------|----------|--------|
| **记忆类型** | 文件型（Markdown） | 无专门记忆系统 | 文件型 + 插件型 |
| **存储位置** | `~/.claude/projects/.../memory/` | SQLite（仅对话历史） | `~/.hermes/memories/` + SQLite |
| **跨会话记忆** | ✅ 自动提取 + 手动编辑 | ❌ 只有单会话历史 | ✅ 内置文件 + 可选插件 |
| **团队共享** | ✅ Team memory | ❌ | ❌ |
| **用户命令** | `/memory`、`/remember`、`/dream` | 无 | `/compress`（带 guided focus） |
| **后台整理** | ✅ Auto-dream（夜间） | ❌ | ❌ |
| **语义搜索** | ✅ 按查询找相关记忆 | ❌ | ✅ 外部插件提供 |

## 二、Claude Code — 多层文件型记忆系统

Claude Code 的记忆系统最复杂，全部基于 **Markdown 文件**，没有数据库。核心理念是：**人类可读的持久化笔记**。

### 2.1 记忆的七层架构（从上到下）

```
第1层：Managed memory    ← /etc/claude-code/CLAUDE.md（系统级，全用户共享）
第2层：User memory        ← ~/.claude/CLAUDE.md（用户级，跨项目）
第3层：Project memory     ← ./CLAUDE.md、.claude/CLAUDE.md（项目级，可git提交）
第4层：Local memory       ← ./CLAUDE.local.md（项目级，不提交git）
第5层：Auto-memory        ← ~/.claude/projects/{项目}/memory/（自动提取的笔记）
第6层：Team memory        ← .../memory/team/（团队共享）
第7层：Agent memory       ← ~/.claude/agent-memory/{类型}/（特定Agent的记忆）
```

### 2.2 核心逻辑流程

#### 【流程1】启动时加载记忆

```
用户启动 Claude Code
    ↓
系统扫描以下位置的指令文件：
  - /etc/claude-code/CLAUDE.md（如有）
  - ~/.claude/CLAUDE.md（用户级）
  - ./CLAUDE.md（项目根目录）
  - .claude/CLAUDE.md（项目子目录）
  - ./CLAUDE.local.md（本地私有）
    ↓
同时加载 auto-memory 入口文件：
  ~/.claude/projects/{项目名}/memory/MEMORY.md
    ↓
所有内容合并注入 System Prompt（系统提示词）
    ↓
模型在后续对话中"知道"这些背景信息
```

> **关键设计**：`CLAUDE.md` 是**指令文件**（告诉AI怎么做事），`MEMORY.md` 是**记忆索引**（记录学到的信息）。两者都走文件系统，但用途不同。

#### 【流程2】对话中自动提取记忆

```
用户和 Claude Code 对话...
    ↓
每轮对话结束后，后台触发【记忆提取Agent】
    ↓
提取Agent分析最近 N 条消息
    ↓
如果主Agent本轮已经写过记忆 → 跳过（避免重复）
    ↓
提取Agent判断哪些信息值得记住：
  - 用户的角色/偏好（user 类型）
  - 用户的纠正/确认（feedback 类型）
  - 项目状态/决策（project 类型）
  - 外部资源指针（reference 类型）
    ↓
写入到 ~/.claude/projects/{项目}/memory/ 目录：
  - 每条记忆一个独立的 .md 文件
  - 文件包含 YAML  frontmatter（name/description/type）
  - 同时更新 MEMORY.md 索引文件
```

> **四种记忆类型**：
> - `user`：关于用户的信息（角色、偏好、知识背景）
> - `feedback`：用户的纠正和指导（"别用mock"、"不要总结"）
> - `project`：项目相关的状态和决策（"周四冻结代码"、"legal要求改auth"）
> - `reference`：外部资源指针（"bug在Linear项目INGEST中跟踪"）

#### 【流程3】查询时召回相关记忆

```
用户输入新消息
    ↓
系统启动【相关记忆预取】
    ↓
用轻量级模型（Sonnet）做 side-query：
  "根据用户的问题，从以下记忆文件中选最多5个最相关的"
    ↓
排除本轮已经展示过的记忆（避免重复）
    ↓
读取选中的记忆文件（最多200行 / 4KB）
    ↓
以 <system-reminder> 形式注入到当前上下文
    ↓
主模型在回复时能看到这些相关记忆
```

> ** freshness 追踪**：每条记忆记录修改时间。超过1天的记忆会带警告："此记忆已 N 天未更新，请核实后再引用"。

#### 【流程4】夜间自动整理（Auto-Dream）

```
时间门：距离上次整理 ≥ 24 小时
    ↓
会话门：期间产生了 ≥ 5 个新会话
    ↓
锁检查：没有其他进程正在整理（文件锁）
    ↓
启动【整理Agent】
    ↓
读取所有会话记录和现有记忆文件
    ↓
将零散的新记忆合并到主题文件中
    ↓
更新 MEMORY.md 索引
    ↓
删除过时/重复的记忆条目
    ↓
写入整理结果
```

> **作用**：避免记忆文件碎片化。比如10个会话都提到"用bun不用npm"，整理后会合并成一条清晰的 feedback 记忆。

### 2.3 用户如何与记忆交互

| 命令 | 作用 |
|------|------|
| `/memory` | 打开交互式界面，浏览和编辑所有记忆文件 |
| `/remember` | 让AI审查 auto-memory，提议提升到 CLAUDE.md 或 team memory |
| `/dream` | 手动触发记忆整理（类似夜间Auto-Dream） |

### 2.4 记忆文件格式

**单条记忆文件**（如 `feedback_testing.md`）：
```markdown
---
name: 测试必须用真实数据库
description: 禁止使用mock数据库进行测试
type: feedback
---

不要用mock测试数据库。上季度mock测试通过了但生产迁移失败。

**Why:** mock和真实数据库的行为差异掩盖了迁移脚本的问题

**How to apply:** 所有集成测试必须连接真实数据库实例
```

**索引文件 MEMORY.md**（无 frontmatter）：
```markdown
- [用户角色](user_role.md) — 数据科学家，专注可观测性
- [测试策略](feedback_testing.md) — 禁止mock数据库
- [发布冻结](project_release.md) — 3月5日移动端发布分支切割
```

## 三、OpenCode — 无专门记忆系统

OpenCode **没有跨会话的记忆机制**。它的设计哲学是：**每个会话独立，不保留长期记忆**。

### 3.1 有什么

```
会话历史存储
    ↓
SQLite 数据库（~/.config/opencode/storage/）
    ↓
表结构：
  - sessions：会话元数据（ID、模型、时间戳、费用）
  - messages：完整消息历史（角色、内容、工具调用、token数）
  - parts：消息片段（多模态内容拆分存储）
  - todos：Todo任务列表
  - permissions：权限决策记录
    ↓
作用：会话可恢复、可搜索、可分享
```

### 3.2 没有什么

- ❌ 没有 `/remember` 或 `/memory` 命令
- ❌ 没有跨会话的自动记忆提取
- ❌ 没有 project-level 的持久化上下文
- ❌ 没有用户偏好学习

### 3.3 设计取舍

OpenCode 认为：
> "如果一个项目需要长期记忆，应该写在 `opencode.json` 配置里或项目文档里，而不是让AI偷偷记下来。"

这种设计更简单、更可预测，但也意味着：
- 每次新会话都从零开始
- AI不会记住"你喜欢简洁回复"
- 需要用户在项目配置中显式声明偏好

## 四、Hermes — 内置文件记忆 + 插件扩展

Hermes 的记忆系统是**两层架构**：一层内置（简单够用），一层插件（能力无限）。

### 4.1 第一层：内置记忆（Built-in Memory）

#### 存储位置

```
~/.hermes/memories/
  ├── MEMORY.md    ← Agent自己的笔记（上限 ~2200字符）
  └── USER.md      ← 用户画像（上限 ~1375字符）
```

#### 逻辑流程

```
【会话开始时】
    ↓
MemoryStore.load_from_disk() 读取两个文件
    ↓
去重、生成【冻结快照】
    ↓
快照注入 System Prompt（系统提示词）
    ↓
【快照在会话期间不可变】← 关键设计！

【会话进行中】
    ↓
用户说"记住我是后端工程师"
    ↓
Agent调用 memory 工具（action: add）
    ↓
写入到 ~/.hermes/memories/MEMORY.md
    ↓
【但系统提示词里的快照不会更新】
    ↓
Agent通过工具返回告诉用户"已记住"
    ↓
【下次会话】新快照才会包含这条记忆
```

> **为什么冻结快照？** 保持系统提示词前缀稳定，有利于LLM的 prefix cache（前缀缓存），降低token成本。

#### 文件格式

用 `§`（章节符）分隔条目：
```markdown
§
用户是后端工程师，主要用Go和Python
§
项目使用bun作为包管理器，不要用npm
§
用户喜欢简洁回复，不要总结
```

> **安全扫描**：所有写入内容都会扫描 prompt injection（提示注入）、数据外泄模式和隐形Unicode字符。

### 4.2 第二层：外部记忆插件（External Memory Providers）

Hermes 支持 **8 种第三方记忆插件**，但**一次只能启用一个**。

| 插件 | 后端 | 核心能力 |
|------|------|----------|
| **Honcho** | 云端/自托管 | 用户建模、同伴卡片、会话上下文 |
| **Hindsight** | 云端/本地 | 知识图谱、实体解析、反思综合 |
| **Mem0** | 云端(Mem0平台) | 服务端LLM事实提取、语义搜索 |
| **Supermemory** | 云端 | 语义记忆、画像回忆、多容器 |
| **Holographic** | 本地SQLite | HRR代数、信任评分、组合查询 |
| **OpenViking** | 自托管 | 文件系统层级、分层加载 |
| **RetainDB** | 云端 | 混合搜索、7种记忆类型、增量压缩 |
| **ByteRover** | 本地/云同步 | 知识树、压缩前提取 |

#### 插件生命周期

```
用户选择记忆插件（config.yaml 中 memory.provider）
    ↓
【初始化】initialize(session_id)
  → 连接后端、创建资源、预热缓存
    ↓
【每轮前】prefetch(query, session_id)
  → 返回与当前查询相关的记忆上下文
  → 以 <memory-context> 代码块注入提示词
    ↓
【每轮后】sync_turn(user_msg, assistant_msg, session_id)
  → 后台线程持久化本轮对话（不阻塞主循环）
    ↓
【会话切换】on_session_switch(new_session_id)
  → /resume、/branch、/reset 时切换状态
    ↓
【会话结束】on_session_end(messages)
  → 最终提取/刷新
    ↓
【退出】shutdown()
  → 清理队列、关闭连接
```

> **关键设计**：所有写操作都在**后台守护线程**中进行，永不阻塞聊天主循环。

### 4.3 第三层：会话存储（SQLite）

```
~/.hermes/state.db（SQLite，WAL模式）

表结构：
  - sessions：会话元数据（ID、来源、模型、时间、父会话ID支持分支）
  - messages：完整消息历史（角色、内容、工具调用、token数、推理内容）
  - messages_fts / messages_fts_trigram：FTS5全文索引（支持CJK三元组）
  - state_meta：键值存储

功能：
  - session_search 工具搜索所有历史会话
  - 支持Gemini Flash对搜索结果做摘要
  - 应用级重试 + 随机抖动处理写竞争
```

### 4.4 与压缩机制的联动

```
对话进行 → 上下文快满了 → 触发压缩
    ↓
【压缩前】MemoryManager.on_pre_compress(messages)
    ↓
记忆插件有机会在上下文被丢弃前提取关键信息
    ↓
压缩执行 → 旧消息被替换为总结
    ↓
【关键信息已通过插件持久化，不会丢失】
```

## 五、三个系统的核心差异

### 5.1 记忆触发方式

| 系统 | 触发方式 | 特点 |
|------|----------|------|
| **Claude Code** | 主Agent自动写 + 后台提取Agent | 双向：模型主动记 + 后台补漏 |
| **OpenCode** | 无 | 无记忆功能 |
| **Hermes** | 工具调用（memory tool） | 模型主动调用，显式控制 |

### 5.2 记忆更新策略

| 系统 | 更新时机 | 是否即时生效 |
|------|----------|--------------|
| **Claude Code** | 每轮结束后后台写入 | 下轮即可通过预取看到 |
| **OpenCode** | N/A | N/A |
| **Hermes** | 工具调用时写入磁盘 | 当前会话不生效（冻结快照），下次会话生效 |

### 5.3 记忆召回方式

| 系统 | 召回方式 | 智能程度 |
|------|----------|----------|
| **Claude Code** | 按查询用Sonnet选相关文件（最多5个） | 高（语义理解查询意图） |
| **OpenCode** | N/A | N/A |
| **Hermes** | 内置：全部注入；插件：prefetch语义召回 | 取决于插件（Mem0/Hindsight高，文件型低） |

### 5.4 记忆范围

| 系统 | 用户级 | 项目级 | 团队级 | 会话级 |
|------|--------|--------|--------|--------|
| **Claude Code** | ✅ CLAUDE.md | ✅ CLAUDE.md + auto-memory | ✅ team memory | ✅ session memory |
| **OpenCode** | ❌ | ❌ | ❌ | ✅ 仅历史记录 |
| **Hermes** | ✅ USER.md | ✅ MEMORY.md | ❌ | ✅ SQLite + 可选插件 |

## 六、对 paw-ts 的借鉴建议

### 6.1 推荐方案：简化版 Claude Code 模型

考虑到 paw-ts 的定位（个人开发助手，非团队协作），推荐采用**简化版文件型记忆**：

```
第一层：Project 记忆
  → .paw/CLAUDE.md（项目级指令，可git提交）
  → .paw/CLAUDE.local.md（本地私有，不提交）

第二层：Auto 记忆
  → ~/.paw/projects/{项目}/memory/MEMORY.md（索引）
  → ~/.paw/projects/{项目}/memory/*.md（主题文件）

第三层：Session 记忆（可选）
  → 用于上下文压缩时的总结替代
```

### 6.2 最小可行实现（MVP）

**Phase 1：手动记忆**
- 支持 `/memory` 命令打开编辑器
- 读取 `.paw/CLAUDE.md` 注入 system prompt
- 记忆格式：Markdown + YAML frontmatter

**Phase 2：自动提取**
- 每 N 轮后 fork 子 agent 分析对话
- 自动写入到 `~/.paw/projects/{项目}/memory/`
- 支持四种类型（user/feedback/project/reference）

**Phase 3：智能召回**
- 查询时用轻量级模型选择相关记忆
- 以 system-reminder 形式注入

### 6.3 关键设计决策

| 决策 | 建议 | 理由 |
|------|------|------|
| **存储格式** | Markdown 文件 | 人类可读、可编辑、可git管理 |
| **索引方式** | MEMORY.md 单文件索引 | 简单有效，Claude Code验证过 |
| **更新策略** | 后台Agent提取 + 用户手动修正 | 自动为主，人工为辅 |
| **召回方式** | 关键词匹配（初期）→ 语义匹配（后期） | 先简单可用，再智能化 |
| **范围** | 项目级 + 用户级 | 个人工具，无需团队层 |
| **安全** | 扫描prompt injection和敏感信息 | 防止记忆被恶意污染 |

### 6.4 与现有系统的对接

paw-ts 已经有：
- ✅ Session 持久化（FileSystemSessionStore）
- ✅ Checkpoint 系统（文件快照）
- ✅ Skill 系统（可注入上下文）

记忆系统可以：
1. 复用 SessionStore 的存储基础设施
2. 在 checkpoint 之外增加一个 memory 目录
3. 通过 skill 系统注入记忆到 system prompt
