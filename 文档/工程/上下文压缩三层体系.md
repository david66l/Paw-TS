# Paw-TS 上下文压缩三层体系

## 问题

LLM 上下文窗口有限（如 128K tokens）。Agent 跑 20 轮对话后，read_file 的输出、shell 命令结果、模型思考过程塞满窗口。不处理，第 21 轮模型看不到最初指令。

三层分别解决不同粒度的问题：

```
原始上下文（128K 窗口）
│
├─ System Prompt (15K) ──── L3 Protect：永不压缩
├─ Tool Definitions (10K) ── L3 Protect：永不压缩
│
└─ History (100K) ───────── 压缩主战场
    │
    ├─ [Turn 0-3]  ──────── L1 Prune：超大的工具输出 → 磁盘
    ├─ [Turn 4-10] ──────── L2 Compact：整个中段 → LLM 摘要
    └─ [Turn 11-15] ─────── L3 Protect：最近几轮不动
```

---

## L1 Prune（规则层，零 LLM 调用）

**核心思路**：把太大的工具输出踢到磁盘，上下文中只留一个预览引用。

`context-pruner.ts` 是纯规则引擎，两种触发方式：

### 阶段 A：单个工具输出超大

`toolResultExceedsLimits()` 检查每个 `[Tool xxx completed/failed]` 块的内容长度。超过 `DEFAULT_MAX_TOOL_OUTPUT_BYTES`（50KB，参考 Claude Code 阈值）→ 整块持久化。

持久化后上下文中的占位格式：

```xml
<persisted-output>
Output saved to disk (120.5 KB). Full output: /path/to/tool-results/xxx.txt

Preview (first 2000 bytes):
...前 2000 字节的预览...
...
</persisted-output>
```

**关键设计**：保留 2000 字节预览。让 LLM 能判断「这个输出跟我的问题有关吗？需要读完整文件吗？」

### 阶段 B：老旧工具输出数量太多

`collectToolSlots()` → `buildKeepSet()`：只保留最近 5 个可压缩工具的结果在上下文，更旧的驱动到磁盘。

**受保护工具白名单**：`skill`、`web_fetch`、`web_search`、`todo_write` 这四个工具的输出**永远不被驱逐**。web_fetch 的结果是 LLM 做判断的核心依据，驱逐了等于白查。

### 处理流程

`processMessageToolBlocks()` 对每条 user 消息中的每个 `[Tool ...]` 块做检查：

```
isProtectedTool?         → 跳过
isPersistedToolResult?   → 跳过（已经处理过）
evict?（超出最近5个）     → 持久化
oversize?（>50KB）       → 持久化
```

## L2 Compact（LLM 层，仅一次辅助模型调用）

**核心思路**：用便宜模型把对话中段总结成结构化 markdown，替换原始消息。

### 触发条件（四重门控）

`maybeCompactHistory()` 在每轮 `executeTurn()` 中被调用。四个条件**全部**满足才触发：

1. **`shouldCompactHistory(budgetSnapshot)`**：historyUsed > historyBudget × 0.7 - 10000
2. **冷却期**：`compactCooldownTurns === 0`（上次压缩后至少过 5 轮）
3. **熔断器未禁用**：`compactor.isDisabled === false`（连续 3 次失败或连续 2 次低收益后自动禁用）
4. **非 thrashing**：`compactor.shouldSkipDueToThrashing() === false`（最近 2 次压缩节省都 < 15%）

### 三段式分区

`ContextCompactor.determineBoundaries()` 把完整历史切成三段：

```
[Msg 0] ← head（初始 goal）        } Head: 前2条消息，始终逐字保留
[Msg 1] ← head                     }
[Msg 2]  }                         }
 ...     } Middle → 被 LLM 总结     } 中间段：送去压缩
[Msg N-5]}                         }
[Msg N-4]                          } Tail: 末尾 ≈20% token 预算的
 ...                               }   消息，始终逐字保留
[Msg N]                            }
```

**设计理由**：
- Head 保留初始意图：如果不保护第 0-1 条消息，LLM 在摘要中可能丢失最初的 goal
- Tail 保留最近上下文：这是 LLM 当前推理的直接依据，压缩会丢失细节
- Middle 是已被处理过的内容：已经被模型读过、执行过、产生过结果的内容

### 压缩 Agent 调用

`runCompressionAgent()` 不跑完整的 Agent 循环——就是一次 `completeAuxiliaryTask()`，输出 8 个章节的 markdown：

```
## Active Task     ← 当前在做什么
## Goal            ← 最终目标
## Progress        ← 已完成、进行中、阻塞
## Key Decisions   ← 关键决策及原因
## Relevant Files  ← 涉及的文件
## Errors & Fixes  ← 错误和修复方法
## Next Steps      ← 下一步做什么
## Pending Questions ← 待解决问题
```

**为什么用辅助模型而不是主模型？**
- 压缩任务是「总结对话」，不需要复杂推理
- 便宜的模型（如 DeepSeek-Flash、Qwen-Turbo）完全可以胜任
- 不消耗主模型（如 Claude Opus）的 token 配额
- 如果没配置辅助模型，回退到主模型

### 质量门控

压缩完不直接替换，先过三道检查：

1. **结构完整性**：`validateCompressionSummary()` 检查三个必须章节（`## active task`、`## goal`、`## progress`）是否存在且非空
2. **节省阈值**：`meetsCompressionSavingsThreshold()` → 压缩后至少比原始少 15% token
3. **去摘要套摘要**：`stripContextSummaryMessages()` 在压缩前剥离已有的 `[Context Summary]` 消息

### 增量更新模式

如果上一次压缩已经生成了摘要，`buildSummaryPrompt()` 切换为增量模式：

```
## Previous Summary
...已有的摘要内容...

Update the summary with the new conversation below.
Preserve information from the previous summary that is still relevant,
and add new key points from the recent conversation.
```

后续压缩只需追加 Middle 段的新内容，而不是重新总结整个历史。

## L3 Protect（保护层）

L3 不是独立模块，是散落在 `context-policy.ts`、`context-manager.ts`、`context-compactor.ts` 中的保护规则集合。

### 保护内容

1. **System Prompt**：`setSystem()` 设置后永不触达截断逻辑。如果裁了 system，Agent 会忘记自己是谁、能做什么
2. **最近 N 个对话回合**：`tailTurnCount`（默认 3）个回合受保护，以 assistant 消息为回合边界
3. **用户约束消息**：包含「不要」「禁止」「必须」「只修改」等关键词的消息，优先级 120，驱逐体系中最高
4. **初始 goal**：第一条非工具结果的 user 消息，保证模型不忘记目标
5. **注入的记忆 + L2 摘要**：记忆检索结果和摘要消息作为最近上下文自然被 tail 保护覆盖

### 驱逐优先级体系

`messagePriorityScore()` 为每条消息打分：

| 类型 | 优先级 | 说明 |
|------|--------|------|
| USER_CONSTRAINT | 120 | 「不要修改 production 配置」 |
| TOOL_RESULT | 95 | 「git status 返回 master, 3 staged」 |
| USER | 80 | 普通用户指令 |
| ASSISTANT_WITH_THINKING | 60 | 模型输出 + 推理过程 |
| ASSISTANT | 40 | 普通模型输出 |

工具结果随时间衰减：每轮 -2 分，最低 45（仍高于普通 assistant）。

### 驱逐算法

`truncateByBudget()`：
1. 从保护级别 3 逐步降级（3→2→1→0），找最小保护方案
2. 可驱逐消息按优先级分排序（低分优先驱逐，同分选成本高的——一次省更多）
3. 如果还不够，对受保护消息也按优先级降级驱逐
4. **最后一条用户消息永远保护**——模型至少得知道用户问了什么

## 上下文预算分配

三层压缩的上游——决定「什么时候该动手」的机制。

`allocateContextBudget()` 把上下文窗口按比例分为四个池：

| 窗口大小 | System | Tools | History | Reserve |
|----------|--------|-------|---------|---------|
| ≤500K | 12% | 8% | 75% | 5% |
| >500K | 8% | 7% | **85%** | 5% |

**为什么大窗口 history 比例更高？** System Prompt 大小相对固定，额外空间全给对话历史。

**Reserve 5% 的作用**：Token 估算存在误差（chars/4 启发式不算精确），streaming buffer 也需要空间。

**触发阈值**：`compactThreshold = historyBudget × 0.7 - 10000`——不是等到满才压缩，70% 就预警。

## 完整数据流

```
executeTurn() 被调用
  │
  ├─ 1. ctxMgr.prune() ───────────────────── L1 入场
  │     ├─ 遍历所有 user 消息中的 [Tool ...] 块
  │     ├─ >50KB → 写磁盘 + 替换为 <persisted-output>
  │     └─ 超出最近5个 → 驱逐一视同仁
  │
  ├─ 2. measureBudget() ──────────────────── 预算快照
  │     systemUsed/toolsUsed/historyUsed vs budget
  │
  ├─ 3. maybeCompactHistory() ───────────── L2 入场
  │     ├─ shouldCompact? (history > 阈值70%)
  │     ├─ 冷却期? (last compact < 5 turns ago)
  │     ├─ 熔断? (3 次失败或 2 次低收益)
  │     ├─ determineBoundaries() → [Head][Middle][Tail]
  │     ├─ stripContextSummaryMessages() → 去摘要套摘要
  │     ├─ runCompressionAgent() → 8章markdown摘要
  │     ├─ validateCompressionSummary() → 质量门控
  │     ├─ meetsCompressionSavingsThreshold() → ≥15%?
  │     └─ ctxMgr.replaceHistory() → 应用
  │
  ├─ 4. 正常模型调用 ─────────────────────
  │
  └─ 5. ctxMgr.addUser/Assistant → 内部自动截断 L3
        └─ truncateHistory() → 优先级驱逐
```

## 面试叙事模板

> "我们有三层压缩。L1 纯规则，把超过 50KB 的工具输出写入磁盘，上下文里只放路径和 2KB 预览，也驱逐超过 5 轮的旧工具结果。L2 是核心，当历史 token 超过分配的 70% 时，用一个便宜辅助模型把中间段总结成 8 章 markdown。压缩前有冷却期（5 轮）防频繁触发，压缩后有质量门控（必须含 active task/goal/progress 三章，节省至少 15%），失败三次自动熔断。L3 是保护层，system prompt、初始目标、用户约束、最近三个回合不受压缩。如果还是超预算，按优先级体系驱逐——用户约束 120 分最高，普通 assistant 消息 40 分最低，工具结果 95 分但随时间衰减。"
